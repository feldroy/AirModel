## AirModel 0.3.0: Automatic table prefixes for safe database sharing

Multiple Air projects can share a single PostgreSQL database without table name collisions. Every table name now includes a prefix derived from the module where the model is defined, with zero configuration required.

```bash
uv add --upgrade AirModel
```

### What's changed

**Table names are now prefixed by module name.** A `UnicornSighting` model defined in `myapp/models.py` produces a table called `myapp_unicorn_sighting` instead of `unicorn_sighting`. The prefix is the top-level package name, so `myapp.sub.models` also produces `myapp_unicorn_sighting`.

For standalone files with generic names (`main.py`, `app.py`, `models.py`, `server.py`), AirModel reads `[project].name` from `pyproject.toml` and uses that as the prefix. A project named `feldroy.com` gets tables like `feldroy_unicorn_sighting`, with common TLDs stripped and hyphens/dots normalized to underscores.

If no `pyproject.toml` is found, the module name is used as-is.

**This is a breaking change.** Existing tables won't match the new names. To migrate, rename your tables:

```sql
ALTER TABLE "unicorn_sighting" RENAME TO "myapp_unicorn_sighting";
```

Or, if you have many tables, generate the statements from your models:

```python
for model in AirModel.__subclasses__():
    old = model.__name__  # derive old name yourself
    new = model._table_name()
    print(f'ALTER TABLE "{old}" RENAME TO "{new}";')
```
